# 💾 Dev Save Skill (저장하기)

> **핵심 원리:** 세이브 파일(`SAVE_POINT.md`)은 반드시 **프로젝트 폴더 안** (`[Han_Rule]/SAVE_POINT.md`)에 저장됩니다.
> 이렇게 하면 폴더를 USB로 복사하든, 클라우드로 동기화하든, 다른 PC에서 열든 — 세이브 데이터가 항상 코드와 함께 이동합니다.

## 🎯 트리거 워드
- **"저장해줘"**, **"세이브"**, **"저장하기"**, **"오늘은 여기까지"**

## 🛠️ AI가 수행할 절차

### 1단계: 이번 세션 작업 내용 수집
AI는 이번 대화에서 수행한 **모든 작업**을 역추적하여 수집합니다:
- 어떤 파일을 생성/수정/삭제했는가
- 각 파일에서 구체적으로 무엇을 변경했는가 (함수명, 클래스명, 로직 등)
- 어떤 버그를 수정했는가
- 어떤 기능을 추가/변경했는가
- 브라우저에서 테스트한 결과
- 아직 끝나지 않은 작업이 있는가

### 2단계: SAVE_POINT.md 파일 생성/덮어쓰기
아래 **고정 양식**으로 `[Han_Rule]/SAVE_POINT.md` 파일을 생성(또는 덮어쓰기)합니다.
**동시에** `[Han_Rule]/PROJECT_ANALYSIS.md`(프로젝트 분석 보고서)도 최신 상태로 업데이트합니다.

```markdown
# 💾 학급운영도구 — 세이브 포인트

> ⚠️ 이 파일은 AI 개발 도우미가 자동 생성한 작업 상태 기록입니다.
> 새 세션에서 **"동기화해줘"**라고 말하면 이 파일을 읽고 이전 작업을 이어갑니다.

## 📅 세이브 시각
- **날짜/시간:** {YYYY-MM-DD HH:MM (KST)}
- **세션 환경:** {어떤 PC/계정에서 작업했는지, 알 수 있으면}

---

## ✅ 이번 세션에서 완료한 작업
1. {구체적인 작업 내용 — 무엇을 왜 했는지}
2. {구체적인 작업 내용}
3. ...

## 📝 변경된 파일 목록

| 파일 경로 | 변경 유형 | 변경 내용 요약 |
|-----------|----------|---------------|
| `app/app.js` | 수정 | {함수명} 추가, {함수명} 버그 수정 등 |
| `app/app.css` | 수정 | {클래스명} 스타일 추가 |
| `{새 파일}` | 생성 | {목적} |

## 🔧 변경 상세 (핵심 코드 맥락)
> AI가 다음 세션에서 코드를 정확히 이해하기 위한 기술적 맥락입니다.

### {파일명 1}
- **변경 위치:** {줄 번호 범위 또는 함수명}
- **변경 이유:** {왜 이렇게 바꿨는지}
- **핵심 로직:** {간략한 코드 설명 또는 핵심 코드 스니펫}

### {파일명 2}
- ...

## ⏳ 미완성 작업 (다음 세션에서 이어서)
- [ ] {구체적인 TODO — 무엇을 어떻게 해야 하는지}
- [ ] {구체적인 TODO}

## 🐛 알려진 이슈
- {발견했지만 아직 수정하지 못한 버그}
- {주의해야 할 사항}

## 💡 다음 AI에게 전하는 메모
> {이 프로젝트의 특수한 맥락, 선생님의 선호, 주의점 등
> 다음 세션의 AI가 알아야 할 모든 것을 여기에 적습니다.}
> 
> 예: "선생님은 빌드 도구 없이 순수 HTML/JS 구조를 선호합니다."
> 예: "app.js는 IIFE 패턴이므로 모듈 분리 시 전역 객체 패턴을 유지해야 합니다."
> 예: "EXP 스케일이 19에서 100으로 바뀐 상태입니다."
```

### 3단계: PROJECT_ANALYSIS.md 전체 재분석 및 새로 작성 (⚠️ 필수)

> **⚠️ 절대 규칙: 부분 수정이 아닙니다. 매번 프로젝트 전체를 다시 분석하고 처음부터 새로 작성합니다.**

"저장해줘"가 호출될 때마다, AI는 `[Han_Rule]/PROJECT_ANALYSIS.md` 파일을 **삭제하고 새로 만드는 수준으로 전체 재작성**합니다. 절차:

1. **프로젝트 전체 스캔:** `app/app.js`, `app/app.css`, `js/` 폴더, 각종 설정 파일 등 모든 소스를 다시 읽고 현재 상태를 파악합니다.
2. **분석 보고서를 처음부터 새로 작성합니다.** 아래 양식을 반드시 포함:
   - **프로젝트 개요** — 스택, 저장소, 인증, 코드 규모 (현재 줄 수 재측정)
   - **파일 구조 맵** — 현재 실제 파일/폴더 구조 반영
   - **현재 구현된 기능 전체 목록** — 코드를 읽고 실제 구현된 것만 표로 정리
   - **아키텍처 강점과 약점** — 현재 코드 품질 평가
   - **선생님께 확인이 필요한 질문** — 미해결 질문, 새로 생긴 질문
   - **생성된 스킬 파일 목록** — 현재 skills/ 폴더 내용 반영
3. **이전 버전과의 차이가 없더라도 반드시 새로 작성합니다.** — 이전 보고서를 복사하지 않고, 현재 코드를 기준으로 다시 분석합니다.

이 파일은 **"지금 이 순간의 프로젝트 스냅샷"**이며, 다른 PC/계정에서 동기화할 때 AI가 프로젝트를 빠르게 이해하기 위한 핵심 문서입니다.

### 4단계: 저장 완료 브리핑

AI는 저장 후 선생님께 다음과 같이 보고합니다:

> **💾 세이브 완료!**
> - 📁 세이브 포인트: `[Han_Rule]/SAVE_POINT.md`
> - 📊 분석 보고서: `[Han_Rule]/PROJECT_ANALYSIS.md`
> - ✅ 완료 작업: {N}건
> - ⏳ 미완성: {N}건
> - 🐛 이슈: {N}건
>
> 다른 PC나 계정에서 이 폴더를 열고 **"동기화해줘"**라고 말씀하시면 됩니다.

## ⚠️ 주의사항
- SAVE_POINT.md는 **항상 덮어쓰기**합니다 (최신 상태만 유지).
- `firebase-config.js`의 실제 키 값은 세이브 파일에 절대 포함하지 않습니다.
- 코드 스니펫을 기록할 때는 **핵심 맥락만** 간결하게 남깁니다 (전체 코드 복사 금지).
- 변경 이유(Why)를 반드시 기록합니다 — 무엇(What)만으로는 다음 AI가 맥락을 못 잡습니다.
