브라우저에서 직접 호출할 때는 **CORS** 때문에 안 될 수 있어, 그때는 같은 화면의 **「직접 입력」** 으로 메뉴를 적으면 됩니다.

| 항목 | 설명 |
|------|------|
| **NEIS 인증키** | [나이스 오픈API](https://open.neis.go.kr/portal/mainPage.do) 에서 발급하는 **인증키(KEY)** |

| **표준학교코드** | API 파라미터 `SD_SCHUL_CODE` (해당 학교의 **표준학교코드**) |

- **위치**: 같은 **「날씨 · 급식 API 설정」** 안에

| **Authentication** | 선생님은 **이메일/비밀번호**, 학생은 **익명(Anonymous)** 로 “로그인된 사용자” 만들기 |
| **Firestore** | 학급 전체 데이터를 **문서 하나**에 JSON 문자열(`payloadJson`)로 저장·불러오기 |

**Cloud Storage, Realtime Database, Functions 등은 이 코드베이스 기준으로는 사용하지 않습니다.**

---

## 2. `firebase-config.js`가 하는 일

`window.ClassStatusFirebaseConfig`에 **웹 앱 등록 시 받는 공개 설정**이 들어갑니다.

- **`apiKey`, `authDomain`, `projectId` …**  
  Firebase SDK가 프로젝트에 붙을 때 필요합니다. (웹에 노출되는 것이 정상인 설정입니다.)
- **`enabled: true`**  
  `app.js`의 `isFirebaseCloudEnabled()`가 이 값과 `apiKey` 존재 여부를 보고, **클라우드 모드**로 동작할지 정합니다.
- **`teacherFirestoreUid`**  
  Firestore에 있는 학급 DB 문서 경로가 **`users/{이 UID}/classStatus/main`** 이라서, **“어느 선생님 계정 폴더를 열지”**를 고정할 때 씁니다.  
  - 선생님이 로그인하면 `initForUser(현재 로그인한 사람의 uid)`로 **본인 문서**를 씁니다.  
  - 학생은 이메일 계정이 없으므로, **익명 로그인 후** `initForUser(teacherFirestoreUid)`로 **선생님과 같은 JSON**을 읽습니다.

즉, **`teacherFirestoreUid`는 “이 학급 데이터가 저장된 Firestore 경로의 주인 UID”**이고, 콘솔 → Authentication → 해당 이메일 사용자 행의 **UID**와 같아야 합니다.

---

## 3. `firebase-sync.js` — 데이터가 오가는 방식

핵심은 **“DB 전체를 한 덩어리 JSON으로 Firestore에 넣는다”**입니다.

1. **`initForUser(uid)`**  
   - `users/{uid}/classStatus/main` 문서를 **한 번 읽습니다.**  
   - 있으면 `payloadJson`을 파싱해서 `ClassStatusCore.hydrateDb()` → 메모리·`localStorage`에 반영.  
   - 없으면 로컬에 있던 DB가 있으면 그걸 올리고(`uploadPayload`), 없으면 빈 상태로 동기화만 켬.

2. **`afterLocalSave(db)`** (`core.js`의 `saveDb`에서 호출)  
   - 앱에서 데이터가 바뀔 때마다 **2초 디바운스** 후 같은 문서에 `payloadJson` 전체를 **`set`** 으로 다시 올립니다.  
   - 그래서 “학생 한 명 비밀번호만 바꿔도” 결국 **문서 전체가 다시 저장**되는 구조입니다.

3. **`resetSync()`**  
   - 로그아웃 등에서 참조를 끊을 때 Firestore ref와 타이머를 비웁니다.

정리하면, **RDB처럼 테이블을 나누지 않고**, **NoSQL 문서 하나에 앱 상태 전체를 직렬화**하는 방식입니다.

## 4. 인증 흐름 (앱 `app.js` 관점)

### 선생님

1. 이메일/비밀번호로 `signInWithEmailAndPassword`.
2. `onAuthStateChanged`에서 **이메일 사용자**(익명 아님)로 들어오면 `initForUser(user.uid)` → **본인 `users/{내 uid}/...`** 에 연결.
3. 세션은 `role: "teacher"` 로 잡힙니다.

### 학생

1. **익명** `signInAnonymously()` (콘솔에서 익명 로그인 사용 설정 필요).
2. `initForUser(teacherFirestoreUid)` → **선생님과 동일한 `payloadJson`** 을 읽음.
3. 로컬에서 이름·PIN 검증 후 `role: "student"` 세션.

그래서 **Firestore 규칙**이 “본인 `uid`만 읽고 쓴다” 수준이면, 학생(익명 UID)은 **선생님 경로에 쓸 수 없습니다.** 학생이 바꾼 내용까지 클라우드에 올리려면 `FIREBASE_SETUP.md`에 적어 둔 것처럼 **규칙을 완화**하거나, **쓰기를 열지 않고** 선생님이 나중에 동기화하는 식의 트레이드오프가 생깁니다.

### 로그아웃·캐시

- 선생님 로그아웃: 세션·로컬 DB 캐시를 비우는 쪽이 자연스럽고, 코드에서는 학생 로그아웃 때 **DB를 지우지 않도록** 플래그(`__classStatusSkipClearDbOnAuth`)를 둔 적이 있습니다(같은 PC에서 선생님이 다시 쓸 때를 위함).

---

## 5. 보안 규칙 — 두 가지 예시가 공존하는 이유

문서에 **서로 다른 목적**의 예시가 있습니다.

1. **엄격한 예시** (`request.auth.uid == userId`)  
   - “로그인한 사람만 자기 폴더” — **선생님 단독 사용**에는 맞습니다.  
   - 이 상태에서는 **학생(익명)이 선생님 문서에 쓰기 불가** → PIN 변경이 클라우드에 안 갈 수 있음.

2. **완화된 예시** (`userId == 'TEACHER_UID'` 일 때 인증된 사용자에게 쓰기 허용)  
   - 학생도 같은 `payloadJson`을 올릴 수 있게 하려는 **실용적妥協**입니다.  
   - 대신 **악의적인 클라이언트가 그 경로를 덮어쓸 위험**이 커지므로, 문서에도 적힌 대로 **App Check·운영 정책**을 고려하는 편이 좋습니다.

---

## 6. 알아두면 좋은 제한

- **문서 크기** 약 **1MB** — 학급·로그가 매우 커지면 한도에 걸릴 수 있습니다.
- **API 키**는 공개되어도 되지만, **Firestore 규칙**이 실제 접근 통제의 중심입니다.
- **동시 편집**은 “마지막 저장이 이긴다”에 가깝고, 여러 명이 동시에 쓰면 충돌 가능성이 있습니다(단일 JSON 덮어쓰기 구조).

## 7. 로컬 모드와의 관계
